# 구현 가이드

## 1. 구현 목표

이 문서는 `personal/reference.md`의 요구를 실제 구현 단계로 풀어낸다.

이번 구현의 핵심 목표는 아래와 같다.

- reference 스타일에 가까운 레이아웃으로 재구성
- About me 영역에 사진, 이름, 3개 링크 버튼 배치
- 왼쪽에 다층 Obsidian 노트 트리 표시
- 우측 그래프/TOC/Related Notes 제거
- 현재 불필요한 Quartz 기본 파일 정리

즉, 이번 구현은 단순한 기능 추가가 아니라 “사이트 표면 구조와 프로젝트 정리”를 함께 다루는 작업이다.

## 2. 1단계: 기준 문서 반영

가장 먼저 해야 할 일은 `personal/` 문서를 읽고 요구를 코드/계획 수준으로 번역하는 것이다.

특히 이번 구현에서 직접 반영해야 하는 요구는 다음과 같다.

- 참고 사이트: `https://hchautran.github.io/`
- 왼쪽에 Obsidian 노트 트리
- 중앙에 About me
- 사진 포함
- 링크 버튼은 Email, LinkedIn, GitHub 세 개만
- About me 아래에 Notes 단락
- 우측 그래프 제거
- Table of Contents 제거
- Related Notes 제거
- News, Publications, Service는 현재 제외

## 3. 2단계: 설정 구조 재정리

`site.config.ts`는 더 이상 단순 텍스트 설정 파일로만 보면 안 된다.
reference를 따르려면 최소한 아래 내용을 관리해야 한다.

- 이름
- 소개 문구
- 이메일/링크
- 프로필 이미지 경로
- 최상위 섹션 목록
- 중간 폴더 정렬 규칙
- 노트 정렬 규칙
- 향후 확장 섹션 슬롯 정의

즉, `site.config.ts`는 “사이트 콘텐츠/탐색 구조의 중앙 설정판” 역할을 하게 된다.

## 4. 3단계: 프로필 자산 처리

현재 프로필 사진은 루트의 `profile.jpg`로 존재한다.

구현 단계에서는 아래 순서로 정리하는 것이 맞다.

1. `profile.jpg`를 원본 자산으로 인정
2. 사이트가 참조할 정적 자산 경로를 결정
3. 생성 또는 자산 정리 과정에서 그 경로로 이동 또는 복사
4. About me 영역에서는 최종 자산 경로만 사용

이렇게 해야 홈 UI 코드가 루트 파일 위치에 직접 의존하지 않게 된다.

## 5. 4단계: 콘텐츠 생성 파이프라인 일반화

현재 생성 로직은 `Datascience` 기준으로 출발했지만, 앞으로는 아래 형태를 목표로 해야 한다.

### 입력

- `obsidian/` 아래 모든 Markdown
- `obsidian/` 아래 첨부 이미지
- `profile.jpg`

### 출력

- 홈 페이지
- 최상위 섹션 페이지
- 중간 폴더 페이지
- 개별 노트 페이지
- 사이트 자산 경로의 프로필 사진

### 일반화 포인트

- 최상위 섹션 자동 인식 또는 설정 기반 정의
- 깊은 폴더 경로 재귀 처리
- 깊이에 따라 상대 링크 자동 계산
- 중간 폴더에도 인덱스 생성 가능
- 섹션/폴더/노트가 섞여도 탐색 구조 유지

## 6. 5단계: 왼쪽 탐색기 재설계

reference 기준으로 가장 중요한 구현 포인트는 왼쪽 트리다.

탐색기는 아래 조건을 만족해야 한다.

- 여러 최상위 섹션 표시
- 섹션 아래 하위 폴더 표시
- 더 깊은 하위 폴더 표시
- 개별 노트 표시
- 긴 구조에서도 읽기 쉬운 디자인 유지

정렬 전략은 아래 두 층으로 구성하는 것이 좋다.

- 핵심 문서: 명시적 수동 순서
- 나머지 문서/새 폴더: 기본 규칙 정렬

이렇게 해야 `Actor-Critic` 같은 새 하위 폴더가 생겨도 전체 구조가 깨지지 않는다.

## 7. 6단계: 중앙 About me 재설계

중앙 영역은 현재 문서형 랜딩이 아니라 아래 구조로 바뀌어야 한다.

### 필수 요소

- 프로필 사진
- 이름
- 짧은 소개
- Email 버튼
- LinkedIn 버튼
- GitHub 버튼
- Notes 제목 또는 단락

### 디자인 방향

- 지나치게 문서 페이지처럼 보이지 않게 조정
- 하지만 Quartz의 기본 정보 구조는 완전히 깨지 않음
- 소개와 노트 탐색 사이의 연결이 자연스럽게 이어지게 구성

## 8. 7단계: 제거할 UI 정리

reference 기준으로 현재 제거할 UI는 명확하다.

- Graph
- Table of Contents
- Related Notes

그리고 아래는 “지금은 비활성화” 상태로 둔다.

- News
- Publications
- Service

중요한 점은, 단순히 숨기는 수준이 아니라 계획상 “현재 서비스 범위 밖”으로 명확히 정리해야 한다는 것이다.

## 9. 8단계: 불필요한 파일 정리

이번 구현에는 코드 수정뿐 아니라 저장소 정리도 포함된다.

공격적 정리 기준으로 아래 파일군을 검토한다.

### 정리 우선 후보

- `public/`
- `tsconfig.tsbuildinfo`
- `docs/`
- `.github/FUNDING.yml`
- `.github/dependabot.yml`
- `.github/pull_request_template.md`

### 검토 후 유지 가능 파일

- 라이선스
- 최소 README
- 배포 워크플로
- Quartz 실행에 필요한 설정/타입 파일

정리 기준은 “이 프로젝트 운영에 직접 필요한가” 하나로 맞춘다.

## 10. 9단계: 후속 확장 슬롯 남기기

현재 구현에서는 `News`, `Publications`, `Service`를 넣지 않는다.
하지만 이후 확장 가능성은 문서와 구조에 남겨야 한다.

예를 들어 아래처럼 생각하면 된다.

- `About me` 아래 후속 섹션
- 별도 상단 섹션
- 프로필 영역 확장 카드

또한 이 섹션들이 추가될 때는 `Table of Contents` 재도입 여부를 같이 검토한다.

## 11. 10단계: 새 노트/새 섹션/새 하위 폴더 추가 시 대응

### 새 노트 추가

1. `obsidian/` 아래 원하는 위치에 작성
2. 필요 시 설정 파일에 정렬 정보 추가
3. 생성 스크립트 실행
4. 탐색기와 링크 결과 확인

### 새 최상위 섹션 추가

1. 섹션 정의 추가
2. 섹션 설명/정렬 규칙 추가
3. 홈과 왼쪽 탐색기에 반영
4. 섹션 인덱스 생성 확인

### 더 깊은 하위 폴더 추가

1. 재귀 경로 처리 확인
2. 상대 링크 계산 확인
3. 탐색기 접힘/펼침 동작 확인
4. 중간 폴더 인덱스 필요 여부 검토

## 12. 유지보수 시 가장 자주 보는 파일

- 기준 문서: `personal/RULE.md`, `personal/reference.md`
- 사이트 설정: `site.config.ts`
- 생성 파이프라인: `scripts/generate-content.ts`
- 레이아웃: `quartz.layout.ts`
- 탐색/정렬 커스텀: `quartz.custom.ts`
- 스타일: `quartz/styles/custom.scss`
- 배포: `.github/workflows/deploy.yml`
- 계획 문서: `plans/*.md`
