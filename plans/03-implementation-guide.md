# 구현 가이드

## 1. 구현 목표

이 문서는 “이 프로젝트가 실제로 어떻게 구현되었는가”를 단계별로 설명한다.

이 프로젝트를 처음 보는 사람이라면 아래 질문에 답할 수 있어야 한다.

- Quartz는 어떻게 들어왔는가?
- 사이트 정보는 어디에서 바꾸는가?
- 노트는 어떻게 생성되는가?
- 화면 구조는 어디서 바꾸는가?
- 새 노트를 추가할 때 어디를 수정해야 하는가?
- 앞으로 `Datascience` 외 섹션이나 더 깊은 폴더가 생기면 무엇을 바꿔야 하는가?

## 2. 1단계: Quartz 프로젝트 기반 넣기

먼저 Quartz 공식 스캐폴드를 현재 저장소 루트에 가져왔다.

이 단계에서 들어온 주요 파일은 다음과 같다.

- `package.json`
- `quartz.config.ts`
- `quartz.layout.ts`
- `quartz/`
- `quartz/styles/`

이 작업의 목적은 “정적 사이트 생성기 기반”을 만드는 것이다.

직접 처음부터 만드는 대신 Quartz 구조를 가져온 이유는 아래와 같다.

- 이미 검증된 빌드 체인을 그대로 쓸 수 있다.
- Obsidian/Markdown 친화적인 기능이 기본 제공된다.
- 향후 Quartz 업데이트를 따라가기 쉽다.

## 3. 2단계: 프로젝트 전용 설정 파일 만들기

다음으로 `site.config.ts`를 만들었다.

이 파일은 이 프로젝트에서 가장 자주 수정될 가능성이 있는 값을 모아둔 곳이다.

포함된 내용은 다음과 같다.

- 사이트 제목
- 사용자 이름
- 소개 문구
- GitHub / LinkedIn / Email
- 카테고리 정의
- 노트 목록
- 노트 slug
- 카테고리별/노트별 정렬 순서

현재 구현은 `Datascience` 중심으로 값을 적어 두었지만, 계획 차원에서는 이 파일이 장기적으로 아래 역할까지 맡아야 한다.

- 여러 최상위 섹션 정의
- 섹션별 소개 문구 정의
- 중간 하위 폴더 설명 정의
- 깊은 폴더 구조에서 필요한 정렬 규칙 정의

왜 중요한가?

이 파일이 없으면 아래 값들이 여러 파일에 흩어진다.

- 홈 화면 문구
- 사이드바 정렬
- 카테고리 제목
- 생성 스크립트 대상 파일

그렇게 되면 수정할 때 실수가 많아진다.

## 4. 3단계: 콘텐츠 생성 스크립트 구현

이 단계가 가장 중요하다.

파일:

- `scripts/generate-content.ts`

### 이 스크립트가 하는 일

#### 1. 원본과 설정이 일치하는지 검증

실제 원본 파일 목록과 `site.config.ts`에 등록된 노트 목록이 다르면 빌드 전에 실패시킨다.

이 검증이 필요한 이유는 다음과 같다.

- 새 노트를 추가했는데 설정에 안 넣은 경우
- 설정에는 있는데 실제 파일이 없는 경우
- 누락된 노트가 조용히 빠지는 사고를 막기 위해

#### 2. 자산 파일 수집

이미지 파일을 전부 찾아서 basename 충돌이 없는지 검사한다.

왜 basename 충돌을 보나?

Obsidian 이미지 삽입은 보통 파일명으로 참조되기 때문에, 같은 이름의 이미지가 여러 개 있으면 어떤 파일을 써야 하는지 모호해질 수 있다.

#### 3. `content/` 재생성

기존 `content/datascience`를 지우고 다시 만든다.

이 방식의 장점은 다음과 같다.

- 이전 생성물 찌꺼기가 남지 않는다.
- 항상 재현 가능한 상태를 만든다.
- “현재 원본 기준으로 다시 만들기”가 쉬워진다.

장기적으로는 이 단계가 `content/datascience`만 지우는 수준을 넘어서, `obsidian` 아래에서 생성한 전체 사이트 트리를 섹션 단위로 다시 만드는 방향으로 확장되어야 한다.

#### 4. 홈 페이지 생성

`content/index.md`를 코드로 생성한다.

이 파일에는 다음 정보가 들어간다.

- 이름
- 역할 소개
- 자기소개 문단
- 외부 링크
- 카테고리 카드

#### 5. 카테고리 페이지 생성

각 카테고리 폴더에 `index.md`를 만든다.

예:

- `content/datascience/rl/index.md`
- `content/datascience/rnn/index.md`
- `content/datascience/transformer/index.md`

이 파일들은 폴더 설명을 제공하고, Quartz의 folder page가 아래에 노트 목록을 붙일 수 있게 해준다.

향후에는 이 로직을 더 일반화해야 한다.

- 최상위 섹션마다 `index.md` 생성
- 중간 하위 폴더에도 필요 시 `index.md` 생성
- 예를 들어 `Datascience/RL/Actor-Critic`이 생기면 `content/datascience/rl/actor-critic/index.md`도 만들 수 있어야 한다.

#### 6. 개별 노트 생성

원본 노트를 읽어 Quartz용 `.md` 파일로 만든다.

여기서 일어나는 작업은 아래와 같다.

- frontmatter 추가
- 설명 추출
- 위키링크 변환
- 이미지 임베드 변환
- 날짜 반영

그리고 향후에는 아래도 고려해야 한다.

- 더 깊은 상대경로에서 링크 계산
- 중첩 폴더 내부 이미지 경로 계산
- 하위 폴더 이름도 URL slug로 안정적으로 변환

## 5. 4단계: Quartz 설정 조정

파일:

- `quartz.config.ts`

이 파일에서는 아래 내용을 바꿨다.

- 사이트 제목
- locale
- baseUrl
- analytics 끄기
- 무시할 폴더 설정
- 폴더 페이지 정렬 함수 연결
- 불필요한 OG 이미지 생성 제거
- 테마 색상/폰트 조정

특히 중요한 것은 아래 두 개다.

### `baseUrl`

GitHub Pages에서 sitemap, RSS, canonical URL 같은 절대 URL 생성에 영향을 준다.

이번 프로젝트에서는 다음으로 설정했다.

- `starpoem17.github.io`

### `ignorePatterns`

Quartz가 원본 `obsidian/`, `personal/`, `plans/`를 직접 콘텐츠로 읽지 않게 막는다.

이 설정이 없으면 원본 노트와 계획 문서까지 사이트에 섞여 들어갈 수 있다.

## 6. 5단계: 레이아웃과 정렬 커스터마이징

파일:

- `quartz.layout.ts`
- `quartz.custom.ts`

### `quartz.layout.ts`

이 파일은 화면에 어떤 컴포넌트를 어디에 놓을지 정한다.

이번 프로젝트에서는 다음 방향으로 수정했다.

- 왼쪽에 탐색기 유지
- 오른쪽 그래프/백링크 제거
- 푸터 링크를 사용자 링크로 교체
- Explorer를 열려 있는 상태로 시작

### `quartz.custom.ts`

이 파일은 Quartz 기본 코드에 바로 섞지 않고, 프로젝트별 커스텀 로직만 따로 모아둔 파일이다.

이 안에는 다음이 들어 있다.

- Explorer 정렬 함수
- 폴더 페이지 정렬 함수
- 푸터 링크 매핑

이렇게 분리해 둔 이유는 “Quartz 기본 코드”와 “프로젝트 전용 규칙”을 분리하기 위해서다.

## 7. 6단계: 스타일 커스터마이징

파일:

- `quartz/styles/custom.scss`

홈 페이지가 너무 기본 Quartz 문서 페이지처럼 보이지 않도록, 아래 스타일을 추가했다.

- 소개 박스
- 외부 링크 버튼
- 카테고리 카드
- 홈 타이포그래피

하지만 전체 구조는 Quartz의 기본 흐름을 유지한다.

즉, 완전한 새 프론트엔드가 아니라 “가벼운 브랜드 적용” 수준이다.

## 8. 7단계: 배포 자동화

파일:

- `.github/workflows/deploy.yml`

이 워크플로는 다음 순서로 동작한다.

1. 저장소 checkout
2. Node 22 설정
3. `npm ci`
4. `npm run build`
5. `public/` 업로드
6. GitHub Pages 배포

이 구조 덕분에 사용자는 로컬에서 매번 수동으로 `public/`을 관리할 필요가 없다.

## 9. 새 노트를 추가할 때 해야 하는 일

이 프로젝트는 자동화되어 있지만, 새 노트를 추가할 때는 아래 순서를 따라야 한다.

1. `obsidian/Datascience/...` 아래에 원본 노트를 추가한다.
2. `site.config.ts`의 `NOTES` 목록에 새 노트를 추가한다.
3. 해당 카테고리의 순서가 중요하면 정렬 위치를 반영한다.
4. 이미지가 있으면 원본 경로에 함께 둔다.
5. `npm run generate:content`
6. `npm run build`

주의할 점은, 새 노트를 원본 폴더에만 추가하고 `site.config.ts`에 등록하지 않으면 생성 스크립트가 실패하도록 만들어졌다는 점이다.

## 10. `Datascience` 외 폴더가 추가될 때 해야 하는 일

향후 `obsidian/AI`, `obsidian/Math` 같은 새 최상위 폴더가 생기면 아래 순서로 대응하는 것이 좋다.

1. `site.config.ts`에 새 섹션 정의를 추가한다.
2. 섹션 설명과 표시 이름을 정한다.
3. 해당 섹션 아래 노트 목록 또는 자동 스캔 규칙을 추가한다.
4. 홈 화면에 새 섹션 카드 또는 탐색 항목이 나타나게 한다.
5. `npm run generate:content`로 생성 결과를 확인한다.

장기적으로는 이 과정을 수동 등록 중심에서 “섹션 자동 발견 + 필요한 부분만 설정 오버라이드” 구조로 바꾸는 것이 이상적이다.

## 11. 더 깊은 폴더 구조가 추가될 때 해야 하는 일

예를 들어 아래 구조가 생길 수 있다.

- `obsidian/Datascience/RL/Actor-Critic/A2C.md`
- `obsidian/Datascience/RL/Actor-Critic/PPO.md`

이 경우 구현/운영 관점에서 중요한 점은 다음이다.

1. 폴더 깊이를 2단계로 가정하는 코드를 줄여야 한다.
2. 상대 링크 계산은 현재 파일 위치 기준으로 일반화해야 한다.
3. 중간 폴더도 페이지를 가질 수 있게 해야 한다.
4. 정렬은 “수동 순서 + 기본 규칙” 혼합 전략이 필요하다.

즉, 확장 시 핵심 포인트는 “노트 개수 증가”가 아니라 “경로 깊이 일반화”다.

## 12. 이 구현의 유지보수 포인트

유지보수 시 가장 자주 보게 될 파일은 아래다.

- 내용/링크/카테고리 변경: `site.config.ts`
- 변환 규칙 변경: `scripts/generate-content.ts`
- 사이트 구조 변경: `quartz.layout.ts`
- 정렬 규칙 변경: `quartz.custom.ts`
- 테마/스타일 변경: `quartz/styles/custom.scss`
- 배포 변경: `.github/workflows/deploy.yml`

이 구분만 기억하면 프로젝트를 훨씬 쉽게 다룰 수 있다.
